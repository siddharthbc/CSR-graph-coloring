COO (Coordinate) Edge List graph_builder.py → coo_edges

Before building CSR, edges are collected as COO pairs. Each undirected edge (u, v) is stored as two directed arcs: (u→v) and (v→u). This is how csr_graph.py builds the rows[] and cols[] arrays fed into scipy.sparse.coo_array.

Neighbor Lookups — neighbors_of(v) O(1) start, O(degree) iteration

The key operation in graph coloring: "give me all neighbors of vertex v". With CSR this is a simple array slice — no pointer chasing, no hash lookups, purely sequential memory access.

Here's exactly how each part of the simulation maps to the actual Python implementation in your picasso package.

① Construction: COO → CSR

class CSRGraph: def __init__(self, num_vertices, edges): # Each undirected edge → two directed arcs for k, (u, v) in enumerate(edges): rows[k] = u; cols[k] = v # u → v rows[m+k] = v; cols[m+k] = u # v → u # scipy builds COO, then converts to CSR coo = coo_array((data, (rows, cols))) self._csr = coo.tocsr() # Expose the two CSR arrays self.offsets = self._csr.indptr # [n+1] self.adj = self._csr.indices # [nnz]

② Neighbor Lookup

def neighbors_of(self, v): # start = offsets[v], end = offsets[v+1] return range(offsets[v], offsets[v+1]) # Usage in palette_color.py: for idx in csr.neighbors_of(v): neighbor = csr.adj[idx] # actual vertex ID # remove v's color from neighbor's list

③ Degree Query

def degree(self, v): return offsets[v+1] - offsets[v] # O(1) — just subtract two array elements!

Why CSR for Graph Coloring?

🏎️ Cache-friendly: adj[] is a flat array → sequential reads hit L1/L2 cache
📦 Compact: Only 2 arrays vs n separate Python lists
⚡ O(1) neighbor start: offsets[v] gives instant position — no traversal
🔁 Iteration speed: adj[start..end] is a contiguous memory slice
🧮 scipy-backed: C/Fortran inner loops for the COO→CSR conversion
🚫 Tradeoff: Static — can't add/remove edges after construction
