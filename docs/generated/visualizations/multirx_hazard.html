color_swap_x multi-rx hazard — 4 PE walkthrough

Why color swap can't save multi-source broadcast on WSE-2.

The setup

4 PEs in a row. Every PE wants to broadcast. Routing:

  • east_color_0 (orange) — inject color. rx = {RAMP, WEST}, tx = {RAMP, EAST} at interior PEs (for both own-inject and forwarding). color_swap_x on.
  • east_color_1 (green) — forward-only color. rx = {WEST}, tx = {RAMP, EAST} at interior PEs. Wavelets land here after one X-link hop (because of the swap).

At each X-link traversal, color swaps 0↔1. So a wavelet injected on color 0 at PE0 appears as color 1 at PE1's WEST input, color 0 at PE2's WEST input, color 1 at PE3's WEST input.

Why color swap can't fix this

Color swap prevents adjacent-hop wavelets from sharing a color on the wire. A single-source broadcast cleanly alternates 0→1→0→1 as it propagates, so any one PE's WEST input only ever sees a single source of color 0 at a time.
With multiple simultaneous sources, every PE does two things on the inject color:
  1. Pulls from RAMP (its own CE injects a wavelet)
  2. Pulls from WEST (a wavelet from another source, having bounced through an even number of swaps, happens to land here on the same color)
The router has a single rx port per color. Two wavelets, one color, different links, same cycle → architectural error. The fabric reports:

two wavelets of the same color C0 arrived from different links R and W. This violates architecture rules.

Color swap solves between-hop collisions. The remaining source-inject vs far-source-forward collision at a shared injection color is unsolvable with plain rx broadcast routing. Fix options: serialize injections via token, use per-source colors, or inject via a one-hop detour color that never mixes with the forward chain.
