# On-Device Picasso Recursion — Design Note

## Problem

Today the Picasso level loop runs on the **host**, not on the WSE. After each
level the host:

1. Reads `colors[]` back from every PE.
2. Finds vertices marked `-2` (candidate list exhausted at this level).
3. Picks the next palette size `P_{k+1}`.
4. Regenerates fresh T-sized color lists for those vertices via MT19937.
5. Uploads `color_list[]`, `list_len[]`, `runtime_config[1]` to every PE.
6. Calls `start_coloring()` again.

That's a full host↔device round-trip **per level**. On the cycle-accurate
simulator it's a few ms; on the appliance path it's multi-second (each call is
a fresh job submission through `SdkLauncher`). A graph needing ~10 levels pays
~10× that latency before the interesting work even begins.

The per-round BSP loop is already fully autonomous on the WSE; only the
*between-level* step breaks out to the host. That step can move on-device with
a small, well-bounded set of changes.

## Proposal

Extend the kernel so a single `start_coloring()` call runs the **entire
Picasso recursion**, returning only when every vertex is coloured (or when a
compile-time `max_level` is exhausted). The host's job collapses to:

```python
upload(csr, boundary, expected_recv, level0_lists, palette_schedule)
start_coloring()        # full recursion on WSE
readback(colors)        # final colouring
```

Three additions to [csl/pe_program.csl](csl/pe_program.csl):

### 1. On-PE PRNG + list regeneration

Every PE re-seeds lists for its own invalid vertices using a fast on-device
PRNG. `xorshift32` seeded from `(pe_id, global_vertex_ids[v], current_level)`
gives per-(vertex, level) determinism without any cross-PE coordination.

```csl
fn xorshift32(state: *u32) u32 {
  var x = state.*;
  x ^= x << @as(u16, 13);
  x ^= x >> @as(u16, 17);
  x ^= x << @as(u16,  5);
  state.* = x;
  return x;
}

fn regen_invalid_lists() void {
  const new_p = palette_schedule[@as(u16, @as(i16, current_level))];
  runtime_config[1] = new_p;
  const n = local_num_verts[0];
  var v: i16 = 0;
  while (v < max_local_verts) : (v += 1) {
    if (@as(i32, v) >= n) break;
    if (colors[v] != -2) continue;       // only regenerate the dead ones
    colors[v] = -1;                       // revive for the next level

    var rng = @as(u32, @as(u16, global_vertex_ids[v]))
              ^ (@as(u32, @as(u16, @as(i16, current_level))) * 0x9e3779b1);
    const base = @as(u16, v) * @as(u16, max_list_size);
    var i: i16 = 0;
    while (i < max_list_size) : (i += 1) {
      rng = xorshift32(&rng);
      color_list[base + @as(u16, i)] =
        @as(i32, @as(i16, rng % @as(u32, new_p)));
    }
    list_len[v] = max_list_size;
  }
}
```

### 2. Palette schedule as a one-shot upload

`palette_schedule[level]` is uploaded once with the rest of the graph. Typical
rules: `P_k = 2·P_{k-1}` (Picasso default) or `P_k = P_0 + k·ΔP`.

```csl
param max_level: i16;
var palette_schedule: [max_level]i32;
var current_level: i32 = 0;
```

### 3. Two-bit OR-reduce at the barrier

Today the sync barrier OR-reduces a single "has_uncolored" flag into
`sync_buf[0]`. Widen the semantic to two bits:

- `bit 0` = any vertex in state `-1` (uncoloured, will retry this level)
- `bit 1` = any vertex in state `-2` (list-exhausted, needs a new level)

`detect_resolve` packs both bits; the existing OR-reduce chain combines them
across PEs unchanged.

Termination logic in `maybe_proceed`:

```csl
const has_unc = (sync_buf[0] & 1) != 0;
const has_inv = (sync_buf[0] & 2) != 0;

if (has_unc) {
  // still working at this level → run another BSP round
  @activate(detect_resolve_task_id);
} else if (has_inv and current_level + 1 < @as(i32, max_level)) {
  // level converged but some vertices are stuck → advance
  current_level += 1;
  regen_invalid_lists();
  @activate(speculate_task_id);
} else {
  // fully coloured (or hit max_level) → return to host
  @activate(exit_task_id);
}
```

`advance_level` also needs to clear the round-scoped state it currently resets
in `detect_resolve` (relay queues, `remote_recv_color[]`, `data_recv_remaining`,
etc.) — same reset block, just reused.

## What stays the same

- Per-round BSP loop (speculate → send_boundary → recv → barrier →
  detect_resolve) is untouched.
- Per-round OR-reduce chain (c8/c9/c10) unchanged — only the payload widens
  from 1 to 2 bits.
- CSR, boundary tables, `expected_recv[]`, and `done_send_buf[]` are
  level-invariant and stay loaded.
- Level-0 `color_list` / `list_len` are still generated by the host (preserves
  today's MT19937 bit-compat for level 0). Levels ≥ 1 come from the on-device
  PRNG.

## Cost

| Resource | Cost |
|---|---|
| SRAM (per PE) | `palette_schedule` is `max_level × i32` (e.g. 32 × 4 = 128 B); one u32 of RNG state reused across vertices | 
| Code | ~30 lines CSL (PRNG, regen, branching in `maybe_proceed`) + delete the host-side level loop |
| Compile params added | `max_level: i16` |
| Host upload change | one extra small array (`palette_schedule`), and level-0 lists remain the only `color_list` upload |

## Trade-offs

1. **Determinism vs. MT19937 golden.** xorshift32 won't match the host
   reference bit-for-bit at levels ≥ 1, so golden comparison for those levels
   becomes "is it a valid colouring with ≤ reference colours" rather than
   "matches reference exactly". `run_csl_tests.py` already reports
   `CSL_fewer`/`CSL_more` and validates adjacency non-conflict, so this is not
   a capability regression.
2. **`max_level` is a compile-time cap.** Same pattern as today's
   `--max-rounds`. Overshooting it exits with some vertices still `-2`; host
   can fall back to a greedy pass or bump `max_level` + recompile.
3. **Palette schedule is fixed per run.** A truly adaptive schedule (e.g. grow
   faster when many invalids persist) would need a second OR-reduce of
   "invalid count", which is feasible but out of scope for v1.
4. **Debugging.** A multi-second hang is now "one kernel run", so we lose the
   per-level host-side checkpoint. Mitigation: leave
   `run_csl_tests.py --host-recursion` flag for the old path.

## Benefits

- **Host round-trips per graph: O(L) → O(1).** For the appliance path this is
  the dominant latency cost today.
- **Fits naturally on CS-3 hardware** where job submission overhead is the
  biggest user-visible cost.
- **Removes the host from the critical loop** entirely — the PEs own the
  whole algorithm once the data is staged.
- **No change required** for the per-round BSP protocol, so all the existing
  correctness work (overflow FAIL, headroom-aware sizing, round-robin drain,
  Phase 5 self-activation) carries over unchanged.
- **Pairs well with the hardware-filter branch** (see
  [SCALING_REPORT.md](SCALING_REPORT.md) Part 3): HW filter removes software
  relay + `max_relay`; on-device recursion removes the host level loop. Both
  together collapse the round-trip and the per-hop cost simultaneously.

## Plan of work

1. Add `max_level`, `palette_schedule`, `current_level`, `xorshift32`, and
   `regen_invalid_lists` to [csl/pe_program.csl](csl/pe_program.csl) and the
   matching layout param in [csl/layout.csl](csl/layout.csl).
2. In `detect_resolve`, pack both bits into `sync_buf[0]`.
3. Rewire `maybe_proceed` to the three-way branch above; hoist the
   per-round reset block so it can also run on level advance.
4. Host side: upload `palette_schedule`; collapse the Python recursion loop in
   [picasso/run_csl_tests.py](picasso/run_csl_tests.py) to a single
   `start_coloring` call with an optional `--host-recursion` fallback flag.
5. Re-run the 2-PE and 4-PE simulator suites to confirm parity; re-run
   H2_631g on CS-3 hardware to confirm the end-to-end latency drop.

## Open questions

- **Recompile per graph:** `max_level` being a compile param means different
  `max_level` values require separate artifacts. Acceptable (same as other
  compile-time bounds) or worth a runtime cap?
- **Level-0 lists:** keep host-generated (MT19937) for golden compat, or
  switch to on-device for a fully self-contained kernel? Former is less
  disruptive; latter makes the kernel truly standalone.
- **`palette_schedule` source:** hardcode the doubling rule, or keep it
  as a host-uploaded array for flexibility? Array gives more experiment
  surface at negligible cost.
